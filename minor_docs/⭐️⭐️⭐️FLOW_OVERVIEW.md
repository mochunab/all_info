# Insight Hub 전체 플로우 (비개발자용)

## 1. 소스 등록 (링크 저장)

사용자가 URL 입력 → 시스템이 "이 사이트를 어떻게 읽을지" 자동 판단

```
URL 입력 (예: wiseapp.co.kr/insight)
  ↓
URL 최적화 — 크롤링에 더 적합한 URL이 있는지 탐색 (RSS 피드, 블로그 경로 등)
  ↓
자동 감지 파이프라인 — 9단계 분석으로 크롤링 방식 결정
  ├─ RSS 피드 있으면 → RSS로 결정 (가장 안정적)
  ├─ 사이트맵 있으면 → SITEMAP
  ├─ 네이버/카카오 등 알려진 플랫폼 → 전용 방식
  ├─ 일반 HTML 페이지 → STATIC (가장 빠름)
  └─ JS로 콘텐츠 로딩하는 사이트 → SPA (브라우저 시뮬레이션)
  ↓
DB에 소스 저장 (방식 + 셀렉터 정보 포함)
```

**AI API 호출**: 자동 감지 7+8단계에서 위 방법들로 판단이 안 되면 AI(Gemini 2.5 Flash Lite)에게 HTML을 보여주고 "이 사이트 어떤 방식으로 읽어야 해?" 물어봄

---

## 2. 크롤링 (기사 수집)

매일 아침 9시 자동 실행 또는 사용자가 "자료 불러오기" 버튼 클릭

```
등록된 소스 목록 순회
  ↓
robots.txt 체크 — 대상 사이트가 크롤링을 허용하는지 확인
  ├─ 허용 → 계속 진행
  └─ 거부 → 해당 소스 스킵 (에러 아님)
  ↓
각 소스별 결정된 방식으로 기사 목록 수집
  ├─ 최근 30일 이내 기사만 필터
  └─ 소스당 최대 5개
  ↓
기사 본문 추출 → 500자 미리보기 생성
  ↓
DB에 저장 (중복 자동 스킵)
```

크롤링 자체는 AI 호출 없음 — 순수 HTML 파싱
봇 User-Agent(`InsightHub/1.0`)로 신원을 정직하게 표시

### 실패 시 자동 복구

```
주 전략 실패 (예: STATIC 0건)
  ↓
대체 전략 시도 (예: SPA)
  ↓
그래도 실패 → 자동 복구 발동 (3단계)
  ├─ 1순위: LLM 직접 추출 — AI가 HTML을 통째로 읽고 기사를 직접 찾아냄 (셀렉터 불필요)
  ├─ 2순위: 9단계 파이프라인 재실행 → 새 전략/셀렉터 감지 → 재크롤링
  └─ 3순위: SPA 기본 셀렉터로 최종 폴백
```

**LLM 직접 추출**: 셀렉터가 깨져도 AI가 HTML에서 기사 제목·링크·날짜를 직접 읽어냄. Tailwind 클래스 변동 등으로 셀렉터가 무력화되는 사이트에서 특히 유효

## 3. AI 요약 (크롤링 직후 자동 실행)

```
본문이 있는 기사 중 아직 요약 안 된 것 최대 30건 선택
  ↓
5개씩 병렬로 AI에 전달
  ↓
결과: 한국어 제목 + 1줄 요약(80자) + 태그 3개
  ↓
DB 업데이트
```

**AI API 호출**: 기사마다 1회 (Gemini 2.5 Flash Lite). 실패 시 로컬 GPT-4.1-mini로 재시도

---

## 4. AI 소스 추천 (사용자 요청 시)

```
사용자가 "이런 분야의 소스 추천해줘" 요청
  ↓
AI가 관련 URL 최대 5개 추천
  ↓
각 URL을 6단계 검증 (접속 가능? 콘텐츠 있음? 최신?)
  ↓
통과한 URL만 사용자에게 반환
```

**AI API 호출**: 추천 요청 시 1회 — Gemini 2.5 Flash Lite가 실시간 웹 검색으로 관련 URL을 찾아서 추천

---

## AI API 호출 정리

| 시점 | 용도 | 모델 | 빈도 |
|------|------|------|------|
| 소스 등록 | 크롤링 방식 판단 | Gemini 2.5 Flash Lite | 소스당 1회 (필요시만) |
| 크롤링 자동 복구 | HTML에서 기사 직접 추출 | Gemini 2.5 Flash Lite → 4.1-mini fallback | 셀렉터 실패 시만 |
| 크롤링 후 | 기사 요약 | Gemini 2.5 Flash Lite → 4.1-mini fallback | 기사당 1회 |
| 사용자 요청 | 소스 추천 (실시간 웹 검색 포함) | Gemini 2.5 Flash Lite | 요청당 1회 |
