# 원숭이 vs AI 로봇 — 매매 기준서

> 최종 수정일: 2026-03-21
> 코드: `lib/crypto/battle-trader.ts`
> 초기 자본: 각 $100

---

## 배틀 컨셉

밈코인 시장에서 **완전 랜덤 매매(원숭이)**와 **시그널 기반 매매(로봇)** 중 누가 이기는지 실제 가격 데이터로 검증하는 실험.

- 30분마다 크론 실행 (크롤링 파이프라인 Phase 5)
- 실제 CoinGecko 가격 기준 PnL 계산
- 부분 청산 지원 (포지션 단위 관리)
- 현금 잔액은 **거래 내역 기반 계산** (스냅샷 순환 참조 버그 방지)
- 포지션 청산은 **atomic update** (`remaining_size > 0.01` 조건) — 중복 매도 방지

---

## 원숭이 (Random Monkey)

### 철학
> "다트 던지기". 아무 코인이나 사서 시간 되면 판다.
> 밈코인 시장 자체의 수익률 벤치마크 역할.

### 진입 규칙

| 항목 | 규칙 |
|------|------|
| 판단 주기 | 30분마다 |
| 진입 확률 | **50%** (동전 던지기) |
| 코인 선택 | `crypto_coins` 활성 코인 중 **완전 랜덤** |
| 포지션 크기 | **$100 기준 5~15%** (랜덤, `STARTING_BALANCE` 기준 고정) |
| 최대 동시 포지션 | **5개** |

### 청산 규칙

| 조건 | 기준 |
|------|------|
| 홀드 기간 만료 | 진입 시 **1시간~7일** 중 랜덤 지정, 도달 시 자동 청산 |

- 손절/이익실현 없음
- 시그널 확인 없음
- 오직 시간만이 청산 조건

## AI 로봇 (Signal Robot)

### 철학
> "데이터가 말할 때만 산다. 데이터가 꺾이면 즉시 판다."
> 데이터 부족 환경에서도 동작하도록 임계값 완화. 분산 투자로 리스크 관리.

### 진입 규칙

#### 시그널 조회 (타임 윈도우 폴백)

`24h → 6h → 1h` 순서로 시그널 조회. 긴 윈도우에서 데이터가 있으면 사용, 없으면 짧은 윈도우로 폴백.

#### 필수 조건 (모두 충족해야 진입)

| 조건 | 기준 | 근거 |
|------|------|------|
| 시그널 라벨 | `extremely_hot`, `hot`, 또는 `warm` | Heat 스케일 기준 모멘텀 확인 |
| weighted_score | **≥ 30** | 데이터 부족 환경 대응 (기존 60 → 완화) |
| 멘션 수 | **≥ 3** | 최소 신뢰도 (기존 5 → 완화) |
| 컨피던스 | **≥ 55** (아래 공식) | 복합 지표 충족 |
| 중복 진입 | 같은 코인 중복 불가 | 한 코인에 집중 위험 방지 |

#### 컨피던스 점수 계산

```
confidence = 50 (기본)
  + 15  if weighted_score >= 50     (강한 시그널)
  + 15  if mention_velocity > 0.5   (버즈 급증)
  + 10  if avg_sentiment > 0.2      (bullish 컨센서스)
  + 10  if fomo_avg > 0.3           (FOMO 감지)
─────
최대 100, 최소 충족 기준 = 55
```

→ 기본(50)만으로는 부족. **score, velocity, sentiment, fomo 중 최소 1개** 보너스가 필요.

#### 포지션 관리

| 항목 | 규칙 | 근거 |
|------|------|------|
| 포지션 크기 | 포트폴리오의 **12%** | 분산 투자 (기존 25% → 축소) |
| 최대 동시 포지션 | **5개** | 최대 60% 투자 (기존 3개 → 확대) |
| 최소 현금 보유 | 포트폴리오의 **20%** | 기회 대기 + 리스크 버퍼 (기존 30% → 완화) |

### 청산 규칙

#### 가격 기반 청산

| 단계 | 조건 | 액션 |
|------|------|------|
| **손절** | 현재가 ≤ 진입가 × **0.90** (-10%) | **전량 청산** |
| **TP1 후 손절** | TP1 이후 현재가 ≤ **진입가** (본전) | **잔량 전부 청산** |
| **TP1** | PnL ≥ **+20%** | 포지션의 **1/3 청산** + 손절을 진입가로 이동 |
| **TP2** | PnL ≥ **+40%** | 포지션의 **1/3 청산** |

#### 시그널 기반 청산 (24h 윈도우 기준 — 단기 노이즈 방지)

| 조건 | 기준 | 이유 |
|------|------|------|
| **시그널 반전** | 24h 라벨이 `cold` | 모멘텀 소멸 |
| **센티먼트 급락** | 진입 대비 sentiment **-0.5 이상 하락** | 커뮤니티 분위기 전환 |
| **velocity 소멸** | mention_velocity **< 0.05** | 더 이상 화제가 아님 |

→ 이 3개 중 **하나라도** 해당되면 즉시 잔량 전부 청산.

## 비교 요약

| | 원숭이 | 로봇 |
|---|---|---|
| 진입 빈도 | 30분마다 50% | 30분마다 조건부 |
| 코인 선택 | 랜덤 | 시그널 최고 점수 (24h→6h→1h 폴백) |
| 포지션 크기 | 5~15% 랜덤 | 12% 고정 |
| 최대 포지션 | 5개 | 5개 |
| 홀드 기간 | 1h~7d 랜덤 | 시그널 유지되는 한 무제한 |
| 손절 | 없음 | -10% (TP1 후 본전) |
| 이익 실현 | 없음 | +20% / +40% / 시그널 반전 |
| 현금 관리 | 없음 | 최소 20% 유지 |

## 데이터 흐름

```
30분 크론 → Phase 5 (battle)
  │
  ├─ 원숭이
  │   ├─ 오픈 포지션 확인 → hold_until 만료? → 청산
  │   └─ 50% 확률 → 랜덤 코인 매수 (5~15%, 1h~7d 홀드)
  │
  └─ 로봇
      ├─ 오픈 포지션 확인
      │   ├─ 현재가 vs 손절 라인 → -10% 이하? → 전량 청산
      │   ├─ TP1 (+20%) 도달? → 1/3 청산 + 손절→진입가
      │   ├─ TP2 (+40%) 도달? → 1/3 청산
      │   └─ 시그널 반전/센티먼트 급락/velocity 소멸? → 잔량 청산
      │
      └─ 시그널 평가 (24h→6h→1h 폴백)
          ├─ extremely_hot/hot/warm + score≥30 + mention≥3
          ├─ confidence ≥ 55 (score/velocity/sentiment/fomo 보너스)
          ├─ 현금 20% 이상 유지 가능?
          └─ 조건 충족 → 12% 매수 + 시그널 스냅샷 저장
```

## 현금 계산 (거래 기반)

```
cash = $100
  - Σ(모든 buy의 trade_size)
  + Σ(모든 sell의 trade_size + pnl)
```

스냅샷 순환 참조 대신 `battle_trades` 테이블에서 직접 계산. 포트폴리오 가치 = cash + Σ(보유 포지션 × 현재가/진입가).

## 중복 매도 방지

`closePosition`에서 **atomic update** 사용:

```sql
UPDATE battle_positions
SET status='closed', remaining_size=0, ...
WHERE id = ? AND remaining_size > 0.01
RETURNING id
```

- 반환 0건 = 이미 다른 호출이 청산함 → trade insert 스킵
- `lazyEvaluateExits`는 API GET에서 제거 (side-effect 금지, race condition 방지)
- 청산은 크롤 파이프라인(`executeBattle`)에서만 실행

| 테이블 | 용도 |
